# FAMS 개발 규칙

## 상환/수령 스케줄 기능 개발 규칙

### 문서화 의무

**새로운 기능/프로세스를 개발할 때마다:**

1. 코드 개발 완료
2. **`REPAYMENT_SCHEDULE.md` 업데이트** (필수)
3. 코드 + 문서 함께 커밋/푸시

### 문서 업데이트 항목

각 기능별로 다음 항목을 추가해야 합니다:

#### 필수 항목
- **기능 설명**: 무엇을 하는가 (목적, 범위)
- **프로세스 플로우**: 단계별 동작 (ASCII 다이어그램 포함)
- **API 라우트**: 엔드포인트 + 요청/응답 JSON 예시
- **규칙**: 
  - 입력 검증 규칙
  - 에러 처리 규칙
  - 표시/동작 로직

#### 선택 항목 (필요시)
- 새 데이터 모델 (테이블/컬럼)
- 새 함수/유틸
- 성능 최적화

### 예시

**카드 결제대금 크롤링 기능 추가 시:**

```markdown
## 카드 결제대금 (2차)

### 개요
재무회계팀 카드 결제대금 자동 크롤링 및 관리

### 데이터 모델
...CardPayment 테이블...

### 크롤링 프로세스
[플로우 다이어그램]

### API 라우트
- POST /api/card-payment/sync
- GET /api/card-payment

### 규칙
- 크롤링 주기: 매일 오전 9시
- 중복 제거: 카드번호 + 결제일 기준
- ...
```

### 커밋 메시지 규칙

```bash
git add .  # 코드 + REPAYMENT_SCHEDULE.md
git commit -m "기능: 설명

- 구현 내용 1
- 구현 내용 2
- REPAYMENT_SCHEDULE.md 업데이트: 섹션명 추가

Co-Authored-By: Claude Sonnet 4.6 <noreply@anthropic.com>
"
```

### Why?

1. **코드-문서 싱크**: 항상 일치하는 최신 정보
2. **유지보수 용이**: 팀원이 기능을 빠르게 이해
3. **확장성**: 다음 개발자가 기능을 개선/확장할 때 현황 파악 가능
4. **감사**: 어떤 기능이 언제 어떻게 구현됐는지 기록
